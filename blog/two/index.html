Learning from Rapid Development: Building a mini LoL stats App in 5 hours

November 21, 2017   

Okay, it wasn’t exactly a strict 5 hours, but I spent around 3 hours prior to development, planning and collecting resources to use for development.

The goal of this challenge was to build a league of legends stats app using only my full stack Javascript knowledge (fun!)

Hours 1, 2, and 3

After receiving the task, I outlined the 3 main things I needed to research:

  1. Riot Games API
  2. NodeJS and ReactJS boilerplates
  3. Which project architecture would suit this project

The Riot API

One of the things I enjoyed about this project was being able to take the time to read up on Riot’s API. I’ve always wanted to use it, but I never found the motivation nor the time to look at it myself.

The documentation overall is very straightforward, but collecting the correct data was the slight tricky part.

  • I needed match data from a given summoner. This was provided to us with this amazing endpoint that returned a JSON of A LOT of data from a single game: /lol/match/v3/matches/{matchId}

  • This required a match ID which was provided to us from this API endpoint: /lol/match/v3/matchlists/by-account/{accountId}/recent The thing about this endpoint, was that it returned the last 20 games along with some other miscellaneous information.

  • The endpoint above would only return the recent match data with a given account ID. This ID is provided from this endpoint: /lol/summoner/v3/summoners/by-name/{summonerName}

  • And just when I thought I was done, I forgot that I also needed the champion’s image to display on the results! This app would not have been completed without a beautiful image of Ahri üòè. Also the match API only provided the champion ID, and not the name.

Le Boiler-plate

Next, I had to do some research on boilerplates to give me a head start on development. Specifically, I needed something that would take away the task of deployment and configuration.

Luckily, I ran into the Heroku-cra-node boiler plate. This boilerplate took care of deploying the application to Heroku, and provided a simple and easy to modify structure for the react-ui components.

Project Architecture

When it comes to project structure, I’ve always looked towards the server/client architecture. The overall application will function as a client side UI that interfaces with the Server’s APIs.

Since the boilerplate provided a simple structure for my react-UI code, I decided to focus on the back end structure.

In the server there are 2 folders of interest:

  1. Routes
  2. Services

Within the routes folder, we have the express routes which defined the endpoints that the client would have access to. Finally the controllers within the routes folder would define which services to use.

Inside the service folder, there exist only one service, and this service alone performs all the necessary calls to RIOT’s APIs.

of course, the service itself is broken down into multiple helper functions that each have their own task.

Hours 4, 5, 6, 7, and 8

Alrightyyyyyyy, now for the fun part! Let me break down each hour in three sentences or less:

Hour 4

Super hyped and ready to attack the project. Spent almost all of this hour on designing the back end service and how it will render on the UI. (See React’s documentation on “Thinking in React”).

Hour 5

Still awake and focused on the task at hand. Spent a majority of this hour writing the service to call all of RIOT’s API. Also decided to jump to the react-ui folder and build out the components.

Hour 6

Realize that I’m making more than the alloted 20 requests per second on RIOTs API. Goes back to the drawing board and decided to change MVP to display one match from a list of 20, and allow user in the client side to request a single match.

Hour 7

Realize i’ve spent too much time on the back end and so little with the front end. Decided to switch up the game and focus on the UI (Bad idea)

Hour 7.8

Realize It’s been 4.8 hours since I started. Panic. O God.

Hour 7.9

Compose myself and went back to tackling the UI a bit more to consume the new API. Sudden burst of motivation because I was able to pass mock data from the back end to the client side.

Hour 8: The Final Hour

I honestly just stopped looking at the time and kept working on the UI. (Sorry for the anti-climatic ending).

The good…

The task overall was a really great experience! Not only was I able to use Riot’s API for the first time, but I was able to test myself on developing an application within 5 hours.

What’s even better was that I was able to brush up some “old” technologies like Axios (Promise based HTTP).

… and the bad

Okay, I’ll admit it; I’m really slow when it comes the UI, and sometimes the end result wouldn’t even be close to “beautiful”.

Also the result page was (not pretty at all), and sadly, I was only able to pass mock data through the application.

Refactoring, optimization, and future upgrades to implement

I actually like the idea of a light-weight application to show a user their summoner information. In fact, I was thinking of expanding this application to do a little bit more and add some fun into it!

Thoughts regarding the application for the task, here are some things I would have done differently:

  1. Since I was given a short amount of time, I should have focused more on functionality than UI.
  2. Store the formatted data in a MongoDB database to prevent redundant calls to Riot’s API.

There you have it! Hope you enjoyed the article!

And now, it’s back to climbing that stupid ladder.
